MVC

  • is a pattern in software design used to implement user interfaces, data and controlling logic.
  • it places emphasis on separation between business logic (controller and model) and display (view)
  • this is a form of separation of concerns.

MVC (continued)

  • Model: defines the data that the system should contain, its attributes and relationships
  • View: defines how the data should be displayed to the users.
  • Controller: receives input from the user, received through the view (optionally updates the model or retrieves data from the model) then updates the view with the new information.

AJAX

  • an acronym for Asynchronous JavaScript and XML.
  • is a collection of technologies that are used to make asynchronous connections to servers.

AJAX(continued)

  • although XML is part of the acronym, nowadays JSON is preferred because it is part of JavaScript, simpler and lighter in size thanks to less formatting and structure data.
  • the fact that it is asynchronous means that it is possible to communicate with a server, submit data and update part of a webpage without having to reload the entire webpage.

Why we need AJAX

  • AJAX makes an application more interactive and enables it to load updates without refreshing.
  • Such features improve the user experience in everyday applications:
  • Google Workspace and Microsoft Office 365 apps automatically save changes to a file every time the user makes a change.
  • WhatsApp immediately notifies a user when a new message or post arrives in their inbox.

Javascript Methods for AJAX

  • XMLHttpRequest Object: an older way to add AJAX functionality.
  • fetch: modern, simpler method based on the Promise API.
  • we will focus on fetch.

Asynchronous Programming

  • asynchronous programming is a technique enables a program to start a potentially long running task and continue to be responsive while waiting for the task to complete.

Promise (Introduction)

  • A promise is a substitute or representative for a value that is not yet known.
  • It is a useful concept in asynchronous requests because it allows the developer to associate handlers to an object whose value has not yet been determined.

States of a Promise

  • A Promise is in one of 3 states:
    • pending: initial state.
    • fulfilled: meaning that the operation was completed successfully.
    • rejected: meaning that the operation failed.
  • A pending promise can be fullfilled with a value from the server or rejected with an error message.

fetch (and the Promise API)

  • fetch is based on the Promise API.
  • a Promise includes:
    • code that can take an indeterminate amount of time to return a result.
    • code that waits for a result.

Promises (illustration)

  • How to create a Promise from NodeJS.dev: 
              
                const done = true; 
                const isItDoneYet = new Promise((resolve, reject) => { 
                  if (done) { 
                    const workDone = 'Here is the thing I built'; 
                    resolve(workDone); 
                  } 
                  else { 
                    const why = 'Still working on something else'; 
                    reject(why); 
                  } 
                });

Promises (illustration)

  • then specifies what happens when the promise is fulfilled (success).
  • catch specifies what happens when a promise is rejected. (error)
  • from NodeJS.dev: 
              
                const checkIfItsDone = () => {
                  isItDoneYet
                    .then(ok => {
                      console.log(ok);
                    })
                    .catch(err => {
                      console.error(err);
                    });
                };

Promise API in fetch

  • A fetch promise only rejects when a network error is encountered.
    • For example; if the server is unreachable.
  • from MDN:
    A fetch() promise does not reject on HTTP errors (404, etc.). Instead, a then() handler must check the Response.ok and/or Response.status properties.
